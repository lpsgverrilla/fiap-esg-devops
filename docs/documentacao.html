1. Descrição do Projeto

O projeto Cidades ESGInteligentes é uma plataforma web de monitoramento de indicadores ESG (Environmental, Social, Governance) construída sobre o trabalho acadêmico anterior do grupo (Monitoramento de Emissões de Carbono, MongoDB/NoSQL). A entrega desta disciplina adiciona à aplicação original todo o ciclo DevOps: containerização, orquestração multi-serviço, suíte de testes automatizados, pipeline de CI/CD e deploy automatizado em dois ambientes (staging e produção) em domínios com TLS.

A API REST implementa operações CRUD para as cinco collections ESG (empresas, emissoes, impacto_social, governanca, metas_sustentabilidade), aproveitando a flexibilidade schema-less do MongoDB — cada empresa registra apenas os indicadores que fazem sentido para seu setor.

2. Pipeline CI/CD

Ferramenta

GitHub Actions, utilizando GitHub Environments para escopar secrets por ambiente (o STAGING_DEPLOY_TOKEN só é injetado em jobs vinculados ao environment staging; o PROD_DEPLOY_TOKEN, apenas em jobs de production).

Workflows

2.1 ci.yml – Integração Contínua

Dispara em todo push ou pull request em qualquer branch. Garante que o código nunca regrida antes de chegar às branches protegidas.

  • Job lint-and-test: executa ruff check, ruff format --check e pytest (7 testes cobrindo health, OpenAPI, ciclo CRUD completo em empresas e flexibilidade schema-less em emissões).
  • Job docker-build: usa Buildx com cache GHA para montar a imagem Docker. Falha rápido se o Dockerfile ou dependências quebram. Depende do lint-and-test.

2.2 deploy.yml – Entrega Contínua

Dispara em push nas branches main e develop. Sequência:

  1. Job test: guard-rail — reexecuta a suíte pytest.
  2. Condicionalmente, um dos dois jobs roda:
    • deploy-staging (se branch = develop): POST https://esg-staging.pedrots.com.br/hooks/deploy-fiap-esg-staging com header X-Deploy-Token.
    • deploy-production (se branch = main): POST https://esg.pedrots.com.br/hooks/deploy-fiap-esg-production.
  3. No VPS, adnanh/webhook valida o token e invoca o deploy.sh do ambiente: 
    git fetch + checkout branch correta
docker compose build --pull
docker compose up -d --remove-orphans
  4. O job espera o endpoint /health responder {"status":"ok"} por até 150 s antes de marcar verde.

Fluxo de trabalho recomendado

feature branch → PR → CI verde → merge em develop
                                  ↓
                       deploy automático em staging
                                  ↓
                       validação manual no ambiente
                                  ↓
            PR develop → main → deploy automático em produção

3. Docker – Arquitetura e Estratégia

3.1 Dockerfile (multi-stage)

Imagem Python 3.12-slim em dois estágios:

  • Builder: instala dependências em prefixo isolado (/install), sem sobrepor o site-packages do sistema.
  • Runtime: copia apenas as dependências instaladas e o código-fonte; executa como usuário não-root app; expõe a porta 8000; define HEALTHCHECK interno consultando /health.

Resultado: imagem final ~180 MB, sem toolchain de build, superfície de ataque mínima.

3.2 docker-compose.yml

Dois serviços em rede Docker dedicada, com nomes parametrizados pelo APP_ENV (permite rodar múltiplas instâncias isoladas no mesmo host):

  • api – FastAPI + Uvicorn. Depende do healthcheck do mongo estar verde antes de subir.
  • mongo – MongoDB 7, volume nomeado mongo-data, healthcheck via mongosh ping. O seed inicial é executado automaticamente pela imagem oficial do Mongo ao encontrar o arquivo /docker-entrypoint-initdb.d/01-seed.js, populando as 5 collections com 10 documentos cada na primeira boot.

3.3 Arquitetura no VPS

Internet ─┬─► nginx :443 (TLS) ─► esg.pedrots.com.br         ─► 127.0.0.1:8007 ─► api(prod)    ─► mongo(prod)
          └─► nginx :443 (TLS) ─► esg-staging.pedrots.com.br ─► 127.0.0.1:8006 ─► api(staging) ─► mongo(staging)

                                adnanh/webhook (127.0.0.1:9000)
                                  ├─► /opt/fiap-esg/staging/deploy.sh
                                  └─► /opt/fiap-esg/production/deploy.sh

Cada ambiente tem container, volume e rede Docker isolados — zero cross-contamination entre staging e produção. O certificado TLS é um SAN emitido pelo Let's Encrypt cobrindo os dois domínios, com auto-renovação via systemd timer.

5. Desafios e soluções

Colisão de project name do Docker Compose

Sintoma: o primeiro deploy de staging subia no ambiente de produção sem criar os containers novos — o healthcheck ficou 502 por mais de 2 minutos até dar timeout.

Diagnóstico: como os dois ambientes clonam o repo para diretórios chamados /opt/fiap-esg/staging/app e /opt/fiap-esg/production/app, o Docker Compose derivou o mesmo project name (app) para os dois, tratando-os como o mesmo projeto e reutilizando os containers de produção.

Solução: adicionar name: fiap-esg-${APP_ENV:-local} no topo do docker-compose.yml. Isso garante que cada ambiente seja um project distinto, com containers, rede e volume isolados.

Emissão de certificado TLS para dois subdomínios novos

Sintoma: o nginx não iniciava com os configs finais pois os caminhos de cert ainda não existiam.

Solução: aplicar configs HTTP stub primeiro (só porta 80, com /.well-known/acme-challenge/), rodar certbot certonly --webroot para os dois domínios em uma única chamada (gerando um cert SAN), e só então substituir pelos configs HTTPS finais.

Webhook compartilhado entre vários apps no VPS

O host já rodava outros 3 deploy webhooks (deploy-assistente-sus, deploy-quando-ein, deploy-blog). O padrão existente foi preservado: adicionamos 2 novos hooks em /etc/webhook/hooks.json, cada um validando um X-Deploy-Token independente, e reiniciamos o webhook.service para recarregar a configuração sem afetar os outros deploys.

Seed automático do MongoDB

O trabalho anterior do grupo já trazia o script JS de seed das 5 collections. Em vez de reescrevê-lo em Python, aproveitamos o comportamento nativo da imagem oficial do MongoDB: arquivos .js em /docker-entrypoint-initdb.d/ são executados automaticamente via mongosh contra o database definido em MONGO_INITDB_DATABASE na primeira subida. Apenas removemos a diretiva use <db> (exclusiva do shell interativo) e mantivemos todos os insertMany().
